هو الخيار الموصى به برمجياً لبدء التشغيل التجاري الفوري؛ حيث يوفر إدارة آلية بالكامل للنسخ الاحتياطي وشبكات توصيل سريعة جداً داخل ليبيا، مع انعدام وجود أي مصاريف صيانة تقنية (DevOps Zero).

مخاطر برمجية قريبة / Short-Term Developer Pitfalls (Part I) Section 4

الخطر 1: تباعد حسابات الفاصلة العائمة (Float Precision) مقابل محرك تحقق قاعدة البيانات الفعلي

Risk 4.1: Mathematical Rounding Discrepancy between Frontend Engine & PostgreSQL Trigger
تأثير المشكلة:

يقوم زناد الحماية `prevent_financial_tampering` في قاعدة البيانات بإعادة حساب الدفعة الأولى بدقة فائقة. الواجهة الأمامية تعتمد على معالج الجافا سكربت `financialEngine.ts` الذي يستخدم أرقام الفاصلة العائمة. أي فارق تدوير يتجاوز 5.0 دينار ليبي سيجعل قاعدة البيانات ترفض إدخال المعاملة بالكامل وتتوقف الجلسة.

Technical Impact:

JavaScript floating point calculation errors can slightly skew rounding. If the difference between client-side computed down payments and PostgreSQL precise numeric rules exceeds 5 LYD, the database will throw an anti-tamper exception, resulting in transaction rejection.

SQL Database Anti-Tampering Trigger Schema (Click to View / انقر للعرض) 
-- Re-calculate exact mathematical model in plpgsql to match financialEngine.ts
v_total_murabaha := NEW.car_price * (1 + NEW.margin_rate);
v_max_bank_capacity := LEAST(v_bank_ceiling, v_customer_salary * v_deduction_rate * v_term_months);
v_expected_bank_repayment := LEAST(v_total_murabaha, v_max_bank_capacity);
v_expected_down_payment := v_total_murabaha - v_expected_bank_repayment;

-- Verify that down payment submitted is within 5 LYD buffer
IF ABS(NEW.down_payment - v_expected_down_payment) > 5.0 THEN
    RAISE EXCEPTION 'تلاعب بالقيم المالية مكشوف! القيمة المتوقعة للدفعة الأولى هي % د.ل، وتم تقديم % د.ل',
        v_expected_down_payment, NEW.down_payment;
END IF;

الخطر 2: انحراف التوقيت والتواريخ في تصفية تقويم المتابعات (Timezone shifts)

Risk 4.2: Timezone Discrepancies in CRM Callback & Reminder Queries
تأثير المشكلة:

تخزن الحقول `callback_date` بتوقيت UTC العالمي. بينما تعمل المتصفحات داخل مكاتب ليبيا بنطاق توقيت طرابلس (GMT+2). هذا الفارق الزمني (ساعتين) سيؤدي في نهاية اليوم إلى ظهور المعاملات المجدولة لليوم في تبويب "الاتصالات المتأخرة" أو العكس، مما يشوش على عمل موظفي المعارض.

Technical Impact:

Timestamps are kept in UTC. Tripoli local time is GMT+2. Filtering potential customers scheduled for "Today" will split dates incorrectly near midnight hours if naively queried, causing scheduling confusion and missed callback alerts.

مخاطر برمجية قريبة / Short-Term Developer Pitfalls (Part II) Section 5

الخطر 3: فجوة وذاكرة صلاحيات الأمان والسياسات وعلاقتها برمز الجلسة للمستخدم

Risk 4.3: RLS Policy Caching & Token Freshness Delays on Role Upgrades
تأثير المشكلة:

تعتمد سياسات أمان الأسطر (RLS) في Supabase على قراءة الصلاحية المضمنة في توقيع الـ JWT الأمني. عند ترقية مستخدم من رتبة "موظف" إلى "مدير" عبر الإدارة السحابية، لا يتأثر متصفح المستخدم فوراً، وسيظل معلقاً برتبته القديمة وتظهر له شاشات فارغة أو أخطاء حجب حتى يسجل خروجه يدوياً ويعود من جديد.

Technical Impact:

Client JWTs are cached inside browser storage. When user metadata role changes from 'staff' to 'manager' in the DB, the token's active payload is not refreshed, causing constant database read/write blocking errors for the upgraded user until their next login.

الخطر 4: عدم وجود جدار حماية للمحاولات المتكررة (Rate Limiting) في دوال الدخول العامة

Risk 4.4: Brute-Force Vulnerability inside Public join-with-code Edge Function
تأثير المشكلة:

دالة `join-with-code` هي واجهة عامة مفتوحة للجميع لاستقبال طلبات التسجيل. نظراً لأن الدالة تتجاوز قيود RLS لتأكيد الأكواد وتفعيل الحسابات، يمكن للبوتات والهاكرز استغلالها لإرسال ملايين الطلبات المتكررة لتخمين أكواد الانضمام للمعارض المشتركة واستهلاك سعة حسابات Supabase المدفوعة بالكامل وتجميد الخدمة.

Technical Impact:

The public endpoint lacks rate limits and validation controls. Attackers can flood the endpoint, brute-force search for valid office join codes, trigger admin creation commands, and crash database auth tiers rapidly.

مخاطر المطورين البعيدة / Long-Term Structural Pitfalls Section 6

الخطر 5: غياب آلية تسوية التضارب عند مزامنة البيانات محلياً بعد عودة الإنترنت

Risk 5.1: Offline Fallback Storage Drift & Lack of Conflict Resolution Mechanics
تأثير المشكلة:

توفر المنظومة خيار حفظ متميز محلياً في `localStorage` عند انقطاع الإنترنت لحماية الحسابات والمتابعات. ولكن عند عودة الاتصال لا توجد آلية تسوية تضارب (Conflict Resolution). إذا عدل موظف ملف العميل وهو غير متصل، وعدله محاسب آخر في نفس اللحظة وهو متصل؛ سيقوم النظام بمسح تعديل المحاسب بالكامل وتفقد البيانات تماسكها المالي.

Technical Impact:

When clients go offline, inputs fallback to local storage. Upon reconnection, there is no reconciliation algorithm to compare client vs server timestamps. Submitting out-of-sync edits risk overriding verified financial audits and customer entries.

الخطر 6: تحديات صيانة ملفات التصميم التراكمية (CSS Maintainability Debt)

Risk 5.2: Vanilla CSS Layout Maintainability in Enterprise Scaling
تأثير المشكلة:

تعتمد كفيل بالكامل على ملفات CSS تقليدية مخصصة لبناء جماليات وتأثيرات الواجهة. مع نمو وتوسع شاشات النظام، ستتعقد صيانة آلاف الأسطر التقليدية وسيواجه المطورون تضارباً في أسماء المحددات (Selectors) وضياع التناسق على أحجام الشاشات المختلفة.

Technical Impact:

Relying on pure global Vanilla CSS stylesheets for high-complexity custom components limits visual scalability. Overriding selectors and managing multiple theme variables across global files leads to visual fragmentation as more developers contribute.

Concrete Engineering and Management Mitigations for Launching
👨‍💻 مهام وقائية للمطورين (Developers):

1. توحيد عمليات التقريب: إجبار الواجهة الأمامية على تدوير أرقام الدفعات العشرية لمرتبتين عشريتين عبر دالة `Math.round()` موحدة قبل إرسال البيانات لقاعدة البيانات.
2. تأمين الدوال السحابية: فرض فحص وتأخير المحاولات (Throttling) وتحديد حد تسجيل من نفس عنوان الـ IP لثغرة رمز الانضمام.
3. تحديث جلسة الرمز الأمني: إضافة مراقب فوري (Realtime Listener) في `AuthContext.tsx` يقوم بتنشيط الجلسة `supabase.auth.refreshSession()` تلقائياً للزبائن المرقّين حديثاً.

💼 Operational & Budget Guide (Management):

1. Launch Budget Allocation: Deploy production on the **Supabase Pro Tier ($25/mo)** immediately. This removes the 200 concurrent connection block, adds daily backups, and accommodates up to 100,000 files.
2. Join Code Access Controls: Force Master Admins to generate complex random codes (`KF-X82A-33Q`) and disable the active state immediately after the required personnel joins.
3. Monthly Threat Audits: Review the Master Admin Security dashboard once every 30 days to clear error stacks and audit access attempts.
